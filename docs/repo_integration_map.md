# Repository Integration Map

## Purpose

This document defines how repositories interact at runtime for the Land Feasibility Platform.

This is a governance document. It defines how the system MUST operate and identifies current deviations that must be removed.

## Repository Naming Map

Governance names used in this document:

- `land_intelligence_core` = current workspace repo/runtime `bedrock`
- `land_intel_scraper` = current workspace repo/runtime `zoning_data_scraper`
- `CAD_generate` = current workspace repo/runtime `GIS_lot_layout_optimizer`

## 1. High-Level System Flow

The governed runtime flow is:

```text
UI
↓
Pipeline (Bedrock / land_intelligence_core)
↓
Parcel → Zoning → Layout → Feasibility
```

Operational rule:

- the UI MUST call the pipeline boundary owned by `land_intelligence_core`
- the UI MUST NOT orchestrate stage sequencing itself

## 2. Repo Interaction Map

### Runtime ownership

- `land_intelligence_core` provides orchestration, stage APIs, canonical contracts, and feasibility evaluation
- `land_intel_scraper` provides zoning datasets, district lookup assets, overlays, extraction outputs, and zoning normalization support data
- `CAD_generate` provides the layout engine and layout geometry generation

### Runtime call map

```text
UI
  -> calls land_intelligence_core

land_intelligence_core
  -> loads and normalizes Parcel
  -> calls zoning runtime using land_intel_scraper datasets/services
  -> calls layout runtime in CAD_generate
  -> evaluates feasibility
  -> returns structured pipeline result to UI
```

### Exact repository interaction rules

`land_intelligence_core`:

- CALLS `land_intel_scraper` for zoning-supporting data, dataset-backed lookup helpers, and rule normalization inputs
- CALLS `CAD_generate` for layout generation
- PROVIDES canonical runtime contracts to the rest of the platform
- PROVIDES the only official pipeline execution boundary

`land_intel_scraper`:

- PROVIDES zoning source data, normalized datasets, district and overlay layers, rule extraction outputs, and evidence lineage
- DOES NOT orchestrate the full platform pipeline
- DOES NOT provide the official `ZoningRules` contract boundary to the UI directly

`CAD_generate`:

- PROVIDES layout search, lot generation, road generation, and export-ready geometry
- DOES NOT orchestrate the full pipeline
- DOES NOT define the canonical contract registry

### Data handoff points

- `Parcel` is created in `land_intelligence_core` and passed into the zoning stage
- `ZoningRules` is emitted by `land_intelligence_core` after consuming `land_intel_scraper` datasets and helpers
- `LayoutResult` is generated by `CAD_generate` and returned through `land_intelligence_core` as the canonical layout-stage output
- `FeasibilityResult` is created in `land_intelligence_core` and returned to the UI

## 3. Current Reality Vs Intended Architecture

### Intended architecture

- `land_intelligence_core` is the single orchestrator
- `land_intel_scraper` is the single zoning ingestion and zoning dataset authority
- `CAD_generate` is the single layout engine authority
- the UI calls only the pipeline API exposed by `land_intelligence_core`
- all cross-repo communication occurs through governed APIs, adapters, and protected contracts

### Current reality

- `land_intelligence_core` already exposes the public stage APIs and pipeline API
- `land_intelligence_core` currently uses direct code-level integration with adjacent repos in places instead of strict service boundaries
- `CAD_generate` still contains parcel-facing and planner-facing runtime paths that overlap with the platform orchestration role
- multiple zoning-related data paths and helper layers exist between scraper assets, runtime fallback logic, and Bedrock-side normalization
- the broader workspace still reflects repository-centric evolution rather than a fully enforced platform boundary

### TO BE FIXED

- direct imports across repositories must be replaced with governed service or adapter boundaries where feasible
- duplicate or overlapping pipeline paths must be removed
- multiple zoning source paths must converge to one governed zoning publication path from `land_intel_scraper`
- `CAD_generate` parcel/planner flows must not become a second orchestrator
- UI surfaces must use `land_intelligence_core` as the only official pipeline execution endpoint

## 4. Data Flow Contracts

The protected data flow is:

`Parcel -> ZoningRules -> LayoutResult -> FeasibilityResult`

### `Parcel`

Created in:

- `land_intelligence_core`

Consumed by:

- zoning stage in `land_intelligence_core`
- layout stage in `land_intelligence_core` when passing parcel context into `CAD_generate`
- UI only as a returned canonical record, not as a locally reinvented schema

### `ZoningRules`

Created in:

- `land_intelligence_core` runtime boundary

Source inputs provided by:

- `land_intel_scraper`

Consumed by:

- layout stage in `land_intelligence_core`
- `CAD_generate` through the governed layout call path

### `LayoutResult`

Created in:

- `CAD_generate`

Canonicalized and emitted by:

- `land_intelligence_core`

Consumed by:

- feasibility stage in `land_intelligence_core`
- UI as canonical layout output

### `FeasibilityResult`

Created in:

- `land_intelligence_core`

Consumed by:

- UI
- run storage and evaluation workflows owned by `land_intelligence_core`

## 5. Execution Paths

### Primary path

This is the only intended production path:

```text
UI
-> land_intelligence_core / pipeline boundary
-> Parcel creation and normalization
-> Zoning lookup using land_intel_scraper assets
-> Layout generation in CAD_generate
-> Feasibility evaluation in land_intelligence_core
-> Structured pipeline result returned to UI
```

Rules:

- the primary path is the only path that may be extended for production use
- all new work must strengthen this path rather than create alternates

### Secondary or legacy paths

The following paths are legacy, transitional, or non-authoritative and must be deprecated:

- direct repo-to-repo imports that bypass a governed adapter boundary
- planner or UI flows that reconstruct orchestration outside `land_intelligence_core`
- any zoning path that bypasses the governed publication flow from `land_intel_scraper`
- any layout path that bypasses `CAD_generate` as layout authority
- any alternate end-to-end flow that returns contract-like objects without using the protected pipeline chain

Status:

- these paths may exist today
- they are not target architecture
- they must not be expanded

## 6. Future Target State

The platform MUST converge to the following state:

- API-based or clearly governed service boundaries between repos
- one orchestrator: `PipelineService` in `land_intelligence_core`
- one official zoning publication path from `land_intel_scraper`
- one official layout engine boundary in `CAD_generate`
- one contract authority in `land_intelligence_core`
- UI calling ONLY the pipeline API owned by `land_intelligence_core`

### Target runtime map

```text
UI
  -> Pipeline API (land_intelligence_core)

PipelineService (land_intelligence_core)
  -> Parcel service (land_intelligence_core)
  -> Zoning service consuming published zoning data from land_intel_scraper
  -> Layout service calling CAD_generate
  -> Feasibility service (land_intelligence_core)
  -> returns canonical pipeline result
```

### Non-negotiable target rules

- repo boundaries must follow domain ownership
- orchestration must remain centralized
- contract authority must remain singular
- data handoffs must remain explicit
- duplicate runtime paths must be removed, not documented as acceptable alternatives
